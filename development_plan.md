# 开发计划：机器人个性化智能交互平台 (软件先行版)

**文档版本**: v2.0
**基于技术蓝图**: `complete_project_analysis_v4.md`

## **1. 核心策略：模拟驱动开发 (Mock-Driven Development)**

在等待物理硬件的同时，我们将创建一个“虚拟机器猫”来模拟真实硬件的行为，使得后端和前端可以独立于物理硬件进行开发、联调和测试。这能极大地提高开发效率，缩短项目总周期。

---

## **2. 阶段零：环境搭建与后端基础 (Sprint 0)**

*   **目标**: 搭建完整的后端开发环境，并创建符合 DDD 思想的核心项目结构。
*   **周期**: 约 2-3 天。
*   **任务清单**:
    1.  **【后端】基础项目搭建**:
        *   按照技术蓝图 v4 的规划，初始化 Node.js 项目，并创建 `/src` 下的核心目录结构（`/core`, `/supporting`, `/infrastructure` 等）。
        *   安装所有核心依赖：Koa, `@koa/router`, `koa-bodyparser`, `ws`, Mongoose, `ioredis`, `dotenv`, `winston`。
        *   搭建基础的 Koa 服务器，配置好路由、中间件，并确保能响应一个 `GET /health` 健康检查请求。
        *   配置好与本地 MongoDB 和 Redis 的连接。

---

## **3. 阶段一：核心上下文建模与模拟器开发 (Sprint 1)**

*   **目标**: 开发核心领域的软件模型，并创建一个能模拟真实机器猫行为的“虚拟机器猫”客户端。
*   **周期**: 约 1 周。
*   **任务清单**:
    1.  **【后端】领域建模**:
        *   在 `/core` 目录下，根据 DDD 分析，编写 `RobotPersonality` 和 `Robot` 聚合的纯 JavaScript/TypeScript 类。定义它们的属性和方法（此时方法内可为空）。
        *   定义 `Command`, `State`, `TraitMap` 等核心值对象。
        *   使用 Mongoose 定义 `RobotPersonality` 的 Schema。
    2.  **【后端】机器人控制上下文 (模拟版)**:
        *   实现 `robot-connector.ws.js`，创建一个 WebSocket 服务器，用于接收和管理来自虚拟机器猫的连接。
        *   实现 `robot-repository.in-memory.js`，用于在内存中管理已连接的 `Robot` 实例。
        *   实现 `heartbeat.service.js`，用于与虚拟机器猫进行心跳检测。
    3.  **【模拟器】创建虚拟机器猫客户端**:
        *   **目标**: 创建一个独立的 Node.js 脚本 (`virtual_robot.js`)，它将扮演“机器猫固件”的角色。
        *   **任务**:
            *   该脚本启动后，作为一个 **WebSocket 客户端**，连接到后端的 WebSocket 服务器。
            *   连接成功后，定时（如每 10 秒）向服务器发送一个包含模拟电池电量的 `pong` 消息，以响应服务器的 `ping`。
            *   监听来自服务器的动作指令（如 JSON 字符串 `{"action":"move","direction":"forward"}`）。收到后，在**控制台打印**出来，例如 `console.log('[Virtual Robot] Executing action: move forward')`。
            *   **这个脚本是整个软件先行开发阶段的基石。**

---

## **4. 阶段二：实现个性化与 LLM 交互 (Sprint 2)**

*   **目标**: 在已有模型和模拟器的基础上，完整实现“个性化 -> LLM -> 动作指令”的核心业务逻辑。
*   **周期**: 约 1-2 周。
*   **任务清单**:
    1.  **【后端】机器人个性上下文**: 
        *   完善 `personality.service.js` 和对应的 RESTful API，实现对 `RobotPersonality` 模型的增删改查。
    2.  **【后端】AI 交互上下文**: 
        *   实现 `llm.client.js`，封装对本地 LLM 服务的 API 调用（如果 LLM 未部署，可以先用一个返回固定文本的假数据 `mock` 函数代替）。
        *   实现 `prompt-builder.js` 和 `response-parser.js`。
        *   开发 `interaction.service.js`，串联整个流程。
    3.  **【后端】打通数据流**: 
        *   将 `interaction.service.js` 与 `robot-connector.ws.js` 连接起来。当 `interaction.service.js` 生成动作指令后，它应该能通过 `control.service.js` 将指令下发给**虚拟机器猫**。
    4.  **【联调】后端与模拟器联调**: 
        *   **测试场景**: 
            1.  启动后端服务和 `virtual_robot.js` 模拟器。
            2.  通过 Postman 调用 API 为虚拟机器人设定“个性”。
            3.  通过 Postman 调用另一个 API 模拟一次语音输入。
        *   **验收标准**: 观察到 `virtual_robot.js` 的控制台打印出了由 LLM（或 mock 函数）生成并经过后端解析后下发的动作指令。

---

## **5. 阶段三：前端开发与完整流程联调 (Sprint 3)**

*   **目标**: 开发前端应用，并与后端进行联调，实现一个完整的、可交互的软件系统。
*   **周期**: 约 1-2 周。
*   **任务清单**:
    1.  **【前端】基础搭建**: 
        *   搭建 React 项目，配置好路由、状态管理（Zustand/Redux Toolkit）。
        *   封装 `axios` 和 `WebSocket` 客户端服务。
    2.  **【前端】页面开发**: 
        *   开发“个性设置”页面，提供表单让用户可以创建和修改机器人的名字、种类、性格等。
        *   开发“聊天”页面，用户可以在输入框中输入文本，通过 WebSocket 发送给后端，并能实时展示 LLM 的回复。
        *   开发“状态显示”组件，能实时显示从后端推送的机器人状态（在线/离线、电池电量等，这些数据由我们的**虚拟机器猫**提供）。
    3.  **【联调】前后端完整联调**: 
        *   **测试场景**: 
            1.  同时启动后端服务和 `virtual_robot.js` 模拟器。
            2.  打开前端页面，应该能看到机器人状态为“在线”。
            3.  进入“个性设置”页面，创建一个有个性的机器人。
            4.  进入“聊天”页面，输入一句话，观察聊天记录中是否出现了符合个性的回复。
            5.  同时，观察 `virtual_robot.js` 的控制台是否打印出了对应的动作指令。
        *   **验收标准**: 用户可以在前端完成从“设定个性”到“进行对话”再到“触发虚拟动作”的完整体验闭环。

---

## **6. 硬件到达后的工作 (Sprint 4+)**

当物理硬件到达后，我们的工作将变得非常简单和顺利：

1.  **完成真实固件改造**: 对真实 `catAI.ino` 固件进行改造和烧录，使其能作为 WebSocket 客户端连接后端。
2.  **替换模拟器**: 停止运行 `virtual_robot.js` 脚本，打开真实的机器猫。
3.  **联调与测试**: 由于后端的接口是统一的，理论上真实硬件可以无缝替换掉虚拟模拟器。此时只需进行少量的联调和 bug 修复。
4.  **攻克 TTS**: 开始研究外部 AI 模块，尝试实现语音播报功能。
